## 单司机大规模配送路线规划方案

### 整体架构

```
输入：起点 + 终点 + N个包裹坐标
         ↓
  【第一层】K-means 聚类（N → K个簇）
         ↓
  【第二层】质心间 TSP（确定簇访问顺序）
         ↓
  【第三层】各簇内部并行规划
         ↓
     拼接 + 跨簇 2-opt
         ↓
输出：完整路线序列
```

---

### 各层核心决策

**聚类（第一层）**
- 算法：K-means++，K = max(N/15, N/√300)
- 距离：直接用 Haversine；有河流/高速等障碍先硬切区域，再各区内聚类
- 目标：每簇 10~20 个点，矩阵元素 < 300

**簇间排序（第二层）**
- K 个质心构建 K×K 矩阵（调一次 AWS Matrix API）
- 跑最近邻 + 2-opt，确定簇的访问顺序
- 规模小（K ≈ 15），耗时可忽略

**簇内规划（第三层）**
- 各簇并行执行，互不干扰
- 每簇单独调 AWS Matrix API 拿道路距离
- 跑最近邻 + 2-opt 得到簇内最优顺序

**拼接**
- 按簇顺序，动态找每个簇的最优进出口（枚举 entry/exit 组合，最小化衔接代价）
- 可选：跑一次跨簇 2-opt，消除边界交叉

---

### 距离计算选型

| 阶段 | 用什么 | 原因 |
|---|---|---|
| 聚类 | Haversine | 不需要精确，快 |
| 簇间质心矩阵 | AWS Matrix | K×K 极小，调一次 |
| 簇内距离矩阵 | AWS Matrix | (N/K)² < 300，并行调用 |
| 最终导航输出 | AWS Routes | 拿转弯指引给司机 |

---

### 规模与成本

| 节点数 | K值 | API元素总数 | 适用算法 |
|---|---|---|---|
| < 50 | — | N² < 2500 | 直接单层 2-opt |
| 50~200 | ~15 | K²+(N/K)²×K ≈ 3000 | 三层方案 |
| 200~500 | ~30 | ≈ 6000 | 三层方案 |
| 500+ | ~50 | ≈ 10000 | 三层 + 考虑 OR-Tools |

暴力全矩阵 200 个点需要 40,000 个元素，三层方案压缩到约 3,000，**费用降低 90%+**，速度因并行执行也大幅提升。

---

### 技术选型一句话总结

> **聚类用 Haversine 粗分组 → AWS Matrix 拿真实道路距离 → 2-opt 优化顺序 → 动态进出口拼接**。节点超过 500 或有复杂约束（时间窗、载重）时，直接上 Google OR-Tools 替换手写算法。
